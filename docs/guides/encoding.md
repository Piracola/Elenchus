# 编码规范指南

本文档用于约束 Elenchus 前后端的字符编码处理，避免再次出现中文乱码、错误下载文件名或导入内容解码失败的问题。

## 1. 统一原则

- 所有源码文件统一使用 UTF-8 保存
- 所有前端页面和静态入口统一声明 UTF-8
- 所有后端文本响应统一显式返回 UTF-8 编码
- 所有 JSON、Markdown、日志与运行时快照统一按 UTF-8 写入
- 所有导入链路优先按 UTF-8 解码，必要时提供受控的兼容回退

## 2. 前端规范

### 源码与静态资源

- `frontend/index.html` 必须保留 `<meta charset="UTF-8" />`
- `frontend/src/**/*.ts(x)`、`*.css`、`*.md` 一律以 UTF-8 保存
- 不允许把乱码字面量直接写入组件文案、配置常量或提示文本
- 若发现 `鍒囬`、`杈╄`、`姝ｆ柟` 一类 mojibake 片段，必须视为缺陷处理

### 文本处理

- 前端仅保留最小必要的乱码兜底修复能力
- 新的用户可见文案必须直接写入正确文本，不能依赖运行期替换表“修回来”
- 文件导入优先使用 `TextDecoder('utf-8', { fatal: true })`
- 仅在明确需要兼容历史文件时，才允许增加 `gb18030` 等回退解码

### API 调用

- JSON 请求使用 `Content-Type: application/json`
- 下载接口必须保证文件名和内容编码一致可解析
- 前端导出 Blob 时必须显式声明 `charset=utf-8`

## 3. 后端规范

### HTTP 响应

- `text/*` 响应必须显式带 `charset=utf-8`
- `application/json` 导出接口也统一显式带 `charset=utf-8`
- `Content-Disposition` 中包含中文文件名时，优先使用 `filename*=UTF-8''...`

### 持久化与事件流

- 运行时事件、会话快照、导出文件统一使用 UTF-8 写盘
- `json.dumps(..., ensure_ascii=False)` 只解决转义问题，不等于自动修复乱码源头
- 若字符串在写盘前已经损坏，必须回溯上游来源，而不是仅在读取侧补丁修复

### 上传与导入

- 用户上传文本默认按 UTF-8 / UTF-8 BOM 解码
- 仅为兼容历史中文环境文件时回退到 `gb18030`
- 新增上传接口时，必须在错误信息中明确提示推荐使用 UTF-8 编码

## 4. 排查流程

出现乱码时，按下面顺序排查：

1. 检查源码文件本身是否已保存为错误编码
2. 检查页面入口是否声明 UTF-8
3. 检查接口响应头是否显式包含正确的 `Content-Type`
4. 检查 WebSocket / REST 收到的原始文本是否在服务端就已损坏
5. 检查导入文件是否为 UTF-8 之外的本地编码
6. 检查历史持久化数据是否已经保存成乱码文本

## 5. 提交前检查

- 新增或修改中文文案后，确认编辑器保存编码为 UTF-8
- 检查页面、提示、评分标签、按钮和下载文件名是否正常显示中文
- 检查导入导出链路是否覆盖中文内容
- 运行前端测试与构建，确保编码修复未引入回归
- 修改代码后同步更新相关架构或开发文档
