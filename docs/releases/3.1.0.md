# Elenchus v3.1.0 发行说明

发布日期：2026-03-21

范围说明：本文覆盖提交 `80c95697c2a3baaa7396ca9fc69f026848dc16fb` 之后到当前 `HEAD` 的新增功能与重要体验升级，主要对应 `2026-03-19` 到 `2026-03-21` 期间落地的功能线。

版本定位：`v3.1.0` 是一次明确的“模式扩展 + 会话资料能力扩展”版本。它不是只在 2.0.0 的运行观测基线上做小修小补，而是把 Elenchus 从“可观察的多智能体辩论平台”继续推进为“支持多模式实验、会话级资料池和更强恢复能力的实验运行时”。

这次版本的核心目标可以概括为一句话：

> 在保留 2.0.0 收口成果的前提下，让会话拥有自己的资料池、让运行时支持更可靠的恢复与回放、并正式引入可独立演进的诡辩实验模式。

---

## 一、版本亮点

### 1. 正式引入诡辩实验模式

Elenchus 现在不再只有单一路径的标准辩论流程，而是同时承载两套明确分离的模式：

- `标准辩论模式`
- `诡辩实验模式`

`诡辩实验模式` 不是在标准模式上追加几个开关，而是拥有独立的：

- Prompt 体系
- LangGraph 运行图
- 观察员角色
- 模式产物结构
- 前端视觉提示与界面反馈

该模式的目标不是“判谁赢”，而是让用户观察修辞操控、谬误使用、叙事漂移和对抗性说服是如何发生的。

### 2. 会话资料池从概念变成完整能力

本版本新增了会话级资料库管线，允许每个 session 拥有自己独立的资料池。

当前系统已经支持：

- 上传 `.txt` / `.md` 文档到当前会话
- 为文档生成结构化 `reference_entries`
- 将资料条目接入 `shared_knowledge`
- 在上下文构建阶段检索会话资料
- 删除文档时同步清理资料条目
- 为特定模式自动注入内置参考文档

这意味着“本场辩论依赖什么资料”终于成为会话本身的一部分，而不是只停留在全局配置或临时 prompt 拼接层。

### 3. 会话持久化升级为文件化快照 + JSONL 事件流

会话存储从原来的数据库中心结构，进一步推进为会话目录下的文件化持久化结构：

- `session.json`
- `events.jsonl`
- `rounds/*.json`
- `documents/*.json`
- `reference_entries/*.json`

这样做带来了几个直接收益：

- 会话恢复和回放不再强依赖单表查询
- 每轮结果可以被单独固化与查看
- 文档、资料条目和会话快照保持同域存储
- 导出、调试和离线排查更直接

### 4. 中断恢复、实时流式交互和回放链路继续增强

本版本继续补强了运行时交互能力，使系统更接近“可长时间实验、可中途介入、可事后回看”的状态：

- 更完整的中断恢复链路
- 更稳定的实时 token / 进度流
- 更清晰的运行进度事件
- 更可靠的聊天区自动滚动与消息落点
- 更稳的回放恢复与观察器联动

对最终用户来说，这意味着不管是 live 运行还是 reload 后恢复，会话状态都更连贯。

### 5. 搜索配置和模型自定义参数能力明显增强

本版本补齐了搜索配置与模型自定义参数的前后端接线，已经支持：

- 更完整的搜索配置管理与健康检查
- 搜索服务工厂与 provider 对齐
- 模型自定义参数输入与保存
- 模型温度等参数透传
- 更一致的 Provider / 模型配置体验

这让 Elenchus 在实验不同模型、不同搜索策略和不同参数组合时，更像一个可配置平台，而不是写死参数的演示系统。

## 二、主要新增能力

### 1. 多模式运行时

新增 `DebateMode` 与 `mode_config` 之后，系统正式支持按模式切换运行图与界面行为。

其中 `诡辩实验模式` 的核心规则包括：

- 禁用搜索
- 不启用裁判评分
- 不启用陪审团讨论
- 每轮由 `观察员` 产出观察报告
- 整场结束后生成最终总览报告
- 自动注入内置谬误库作为会话公共资料的一部分

这使得 Elenchus 从“单一标准辩论应用”升级为“共享底座 + 多模式实验链路”的平台结构。

### 2. 会话资料库与内置文档注入

本版本新增的资料库能力，是这段区间里最重要的功能扩展之一。

实现上已经覆盖：

- 文档上传接口
- 会话文档存储
- 文档预处理器
- 参考条目存储
- 参考资料检索服务
- 上下文构建接入

预处理阶段支持两种路径：

- `LLM-first`：优先让模型抽取结构化资料
- `heuristic fallback`：当模型不可用或不适合时退回启发式抽取

此外，`诡辩实验模式` 会自动把内置谬误目录写入当前 session 的：

- `documents/`
- `reference_entries/`
- `shared_knowledge`

这让“用户上传资料”和“系统自动注入资料”首次落在同一套会话级管线里。

### 3. 文件化会话快照与轮次结果固化

在 `v3.1.0` 中，运行时数据不再只是事件流记录，而是形成了更清晰的文件边界：

- `session.json` 保存会话快照、模式配置、模式产物和共享知识
- `events.jsonl` 保存实时运行事件
- `rounds/*.json` 保存每轮固化产物

这个变化对后续能力非常关键，因为它直接支撑了：

- 会话恢复
- 回放重建
- 模式产物查看
- 文档与知识条目同目录管理

对于调试和发行包用户来说，这种布局也更容易直接定位问题或备份单场会话。

### 4. 更强的中断恢复与流式交互

这一轮新增的不只是“继续能跑”，而是让运行时更适合真实实验：

- 新增更细粒度的 `runtime_progress` 事件
- 增强流式调用安全包装与消息修复
- 改善中断后的状态衔接
- 改善会话重新载入后的界面恢复
- 聊天区滚动逻辑更贴近流式输出场景

同时，状态观察器现在会过滤 `pong` 心跳事件，减少无意义事件对回放与观察的干扰。

### 5. 搜索与模型配置能力扩展

这段区间新增的配置能力，主要落在两个方向：

- 搜索系统更可配置
- 模型参数更可实验

现在用户可以在前端更完整地配置：

- 搜索服务参数
- 搜索启用与连接状态
- Provider 模型自定义参数
- 温度等生成参数

后端也同步补齐了相关 schema、服务和测试，使这些参数不只是 UI 展示，而是能真实进入运行链路。

### 6. 开发与启动体验增强

除了用户侧功能，本版本也补齐了开发入口与本地启动体验：

- 新增独立的前后端开发启动脚本
- 新增端口清理脚本
- 优化 `start.ps1`
- 改善前端观察器在不同布局下的停靠逻辑

这使得本地联调和日常启动更稳定，尤其对频繁切换前后端开发的维护者更友好。

## 三、重要体验改进

虽然下面这些改动不一定都属于“全新模块”，但它们明显改变了实际使用感受：

- 修复辩论错误信息中的乱码问题，并补齐前后端文本修复链路
- 改善被拦截请求时的提示显示
- 优化会话恢复后聊天区、观察器和回放状态的一致性
- 设置页支持明文复制 API Key，便于本地配置迁移与排障
- 清理设置页关闭按钮重复等界面问题

这些改动共同提升了系统的“可读性、可恢复性和可操作性”。

## 五、适合哪些用户升级

`v3.1.0` 特别适合下面几类用户：

- 想在同一平台里同时保留标准辩论与实验型辩论链路的开发者

## 六、升级后最值得关注的变化

### 1. 会话目录成为新的核心事实来源之一

升级后，一场 session 的关键状态不再只依赖数据库表，而是会显式落到 `runtime/sessions/<session_id>/` 下的多个文件中。

这会影响：

- 调试方式
- 导出方式
- 故障排查方式
- 后续做迁移脚本或离线工具的思路

### 2. 诡辩实验模式是独立模式，不是标准模式的子配置

如果你之前默认所有辩论都共享同一条 graph，需要注意：

- 诡辩实验模式有独立 graph
- 它不使用标准评分链路
- 它不会启用搜索
- 它更强调“观察报告”而不是“裁判结论”

### 3. 资料来源已经进入会话生命周期

上传文档、内置文档和共享知识之间的关系现在更紧密了。后续无论是做导出、审计还是回放，都应该把它们当作 session 的一部分来看待。

## 八、v3.1.0 的意义

`v3.1.0` 的意义不在于单点功能数量更多，而在于 Elenchus 的平台边界又往前走了一步：

- 它现在不仅能跑辩论，还能按模式运行不同实验链路
- 它现在不仅能保存结果，还能保存会话自己的资料与结构化知识
- 它现在不仅能实时展示过程，还能更可靠地恢复、回放和复盘

如果用一句话总结这个版本：

> Elenchus v3.1.0 在 2.0.0 的运行观测基线上，正式补齐了多模式运行、会话级资料池和文件化持久化三条关键能力线。
