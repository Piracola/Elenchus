# Elenchus 2.0.0 发行说明

发布日期：2026-03-18

版本定位：`2.0.0` 是一次明确的“平台收口”版本。它不是继续把系统做得更大，而是把前一阶段已经堆起来的运行链路、观测界面和文档体系重新整理成更稳定、更可调试、更适合继续演进的基线版本。

这次版本的核心目标只有一句话：

> 保留 LangGraph 作为实验内核，把 Elenchus 收口成一个更稳定的 Agent Runtime Observatory。

---

## 一、版本亮点

### 1. 运行链路显著简化

后端新增了统一的 `RuntimeBus`，把原本分散在多处的运行事件能力收口到一个核心对象中：

- 运行事件封装
- 事件顺序号分配
- 运行事件持久化
- WebSocket 广播
- 连接管理

同时新增了 `DebateRunner` 作为新的运行入口命名，使运行链路更接近我们现在确定的目标结构：

```text
runner + bus + repository
```

这意味着后续排查运行问题时，不再需要在多层抽象之间来回跳转，定位成本明显下降。

### 2. 前端观测区收口为单一观察器

前端不再让多个观测面板同时挤占主聊天区，而是统一收口为一个 `RuntimeInspector`。

`RuntimeInspector` 现在包含三个核心标签页：

- `ExecutionTimeline`
- `LiveGraph`
- `MemoryPanel`

这次调整后，主界面的职责更清晰：

- 左侧：会话与配置
- 中间：辩论消息主区
- 右侧或内联：运行观察区

对于宽屏设备，观察区会稳定停靠在右侧；对于窄屏设备，观察区仍然保持可折叠的内联形态。

### 3. 主路径 UI 更稳定

本次版本重写或清理了以下关键界面：

- `ChatPanel`
- `StatusBanner`
- `ExecutionTimeline`
- `LiveGraph`
- `MemoryPanel`
- `RuntimeInspector`

重点改进包括：

- 顶部状态栏默认折叠，减少屏幕占用
- 聊天主区与观测区职责分离
- 主观测界面的文案与乱码问题集中清理
- 时间线、回放、导入导出、事件定位等核心能力保留

### 4. 记忆溯源能力继续保留

本次版本没有牺牲已有的记忆观测能力，反而把这部分能力纳入了新的收口架构中。

现在系统仍然支持：

- `memory_write` 事件展示
- 记忆来源回跳
- 记忆图谱
- 知识时间线
- 运行事件与聊天消息之间的焦点联动

### 5. 文档体系完成一次清理

旧的重构计划文档已经退出主文档体系，包括：

- `REFACTOR_CHECKLIST.md`
- `REFACTOR_PHASE2_LOG.md`
- `docs/EVENT_DRIVEN_ARCHITECTURE.md`

取而代之的是新的基线文档：

- `PLATFORM_SIMPLIFICATION_PLAN.md`

它不再描述一条冗长的重构路线，而是明确当前版本已经确认的架构边界，后续开发默认以它为准。

## 二、适合哪些用户升级

`2.0.0` 特别适合下面几类用户：

- 已经觉得 1.x 阶段系统越来越复杂、越来越难定位问题的开发者
- 需要一个更稳定的多智能体实验平台，而不是继续堆概念功能的团队
- 想保留 LangGraph，但又希望外围运行层更容易调试、更容易维护的使用者
- 需要可观察的运行过程，而不是只看聊天结果的调试型用户

## 三、主要变化明细

### 后端

- 新增 `backend/app/runtime/bus.py`
- 新增 `backend/app/runtime/runner.py`
- 调整依赖注入，统一切换到 `RuntimeBus` 与 `DebateRunner`
- `EventStreamGateway` 改为兼容包装层
- `ConnectionHub` 改为兼容包装层
- `websocket` 入口改为直接依赖统一运行总线
- 保留并接入现有 memory source tracing 相关改造
- 新增 `backend/tests/test_runtime_bus.py`

### 前端

- 新增 `frontend/src/components/chat/RuntimeInspector.tsx`
- 重构 `frontend/src/components/ChatPanel.tsx`
- 重构 `frontend/src/components/chat/ExecutionTimeline.tsx`
- 重构 `frontend/src/components/chat/LiveGraph.tsx`
- 重构 `frontend/src/components/chat/MemoryPanel.tsx`
- 重构 `frontend/src/components/chat/StatusBanner.tsx`
- 新增 `frontend/src/utils/memoryGraph.ts`

### 文档

- 新增 `PLATFORM_SIMPLIFICATION_PLAN.md`
- 重写 `docs/README.md`
- 删除旧的阶段性重构文档

## 四、与 1.x 相比的变化

### 行为变化

- 主界面不再同时暴露多个并列观测面板
- 观测能力统一进入 `RuntimeInspector`
- 顶部状态栏默认折叠
- 宽屏布局下的观测区改为固定侧栏，不再持续侵入顶部浮层路径

### 架构变化

- 运行事件能力从“网关 + 广播 + 连接”分散结构，收口为统一总线
- 运行入口从旧命名体系向 `DebateRunner` 收敛
- 当前系统正式承认的核心事实源只剩两类：
  - `session snapshot`
  - `runtime events`

### 文档变化

- 旧的“长期大重构路线图”不再作为当前实现标准
- 后续开发应以 `PLATFORM_SIMPLIFICATION_PLAN.md` 为唯一有效基线

## 五、可能影响现有使用方式的地方

### 1. 这是一个带有架构收口性质的版本

如果你之前在本地基于旧的观测布局做过定制，这次升级后需要重新确认你的前端改动是否仍然适配新的 `RuntimeInspector` 结构。

### 2. 旧文档入口已经变化

如果你过去依赖旧的重构计划文档，请改为阅读：

- `PLATFORM_SIMPLIFICATION_PLAN.md`
- `docs/README.md`

### 3. 兼容层仍然保留，但不建议继续依赖旧抽象扩展新功能

`EventStreamGateway` 和 `ConnectionHub` 还在，是为了兼容现有代码，而不是鼓励继续在那条旧链路上扩展。

后续新功能建议直接围绕：

- `RuntimeBus`
- `DebateRunner`
- `SessionRuntimeRepository`

来设计。

---

## 六、升级建议

如果你是从 1.x 升级，建议按下面顺序验证：

1. 启动后端，确认基础 API 与 WebSocket 正常
2. 启动前端，确认 `RuntimeInspector` 可以正常显示
3. 新建一场辩论，确认消息区、时间线、流程图、记忆面板都能正常联动
4. 检查回放、导入、导出是否正常
5. 如你有本地二次开发，重点检查旧的观测面板挂载点是否仍存在

---

## 七、发行包与运行方式说明

`2.0.0` 仍然同时支持两种运行方式：

### 开发模式

适合开发者本地联调。

通常需要：

- Python
- Node.js
- npm

### 轻量发布模式

适合普通用户使用已构建好的前端静态资源。

当前设计下：

- 维护者需要先构建前端静态文件
- 普通用户运行发布包时，默认只需要 Python

此外，`ELENCHUS_ENCRYPTION_KEY` 现在会在首次启动时自动生成并写回本地配置文件，不再要求用户手动创建密钥。

## 九、已知问题

当前版本仍有几个非阻塞问题需要后续继续处理：

- 前端构建仍有 `chunk size warning`，说明包体积偏大，后续可以继续做代码分块
- 仓库里仍然存在一些更早期文件的历史编码问题，虽然主路径界面已经完成主要清理，但并不是全仓所有旧文件都已完全统一
- 这次版本的重点是“收口和稳定”，不是“功能面继续扩张”，因此未来实验型能力会更倾向放进 Inspector，而不是重新进入主聊天区

---

## 十、2.0.0 的意义

`2.0.0` 的意义，不在于功能数量更多，而在于系统开始真正从“不断堆功能的实验项目”，转向“有明确边界、可继续演进的平台基线”。

如果用一句话概括这个版本：

> Elenchus 2.0.0 保留了 LangGraph 的实验性，同时第一次把外围平台结构真正收回到了可维护、可调试、可继续扩展的状态。
