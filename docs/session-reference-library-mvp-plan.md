# 会话级资料池 MVP 规划

> 说明：本文档保留的是**历史规划背景与当时的设计判断**，不是当前实现的唯一事实来源。
> 当前实现请优先参考：[session-reference-library-implementation.md](./session-reference-library-implementation.md)

## 1. 这份文档现在的定位

这份 MVP 文档保留的价值主要是：

- 记录当时为什么要引入“会话级资料池”
- 保留最初的范围约束与取舍判断
- 帮助理解当前实现为何会演变成现在的结构

如果你想确认**现在系统已经怎么做**，请直接阅读实现文档，而不是继续把本页当成当前规格说明。

## 2. 最初目标

本规划的目标，是为每场辩论增加独立资料池，支持用户上传参考文档，由预处理 agent 整理后供辩手参考。

当时定义的 MVP 目标包括：

- 每场辩论可以绑定 0-N 份参考资料
- 资料在进入辩论前先经过预处理，不直接把原文塞进 prompt
- 事实核查 agent 只负责校验可验证声明，不负责文件解析或格式检查
- 辩手读取的是结构化、压缩后的资料，而不是整篇文档
- 不引入向量库或复杂索引系统

当时明确不做：

- 全局跨会话共享资料库
- 基于 embedding 的检索系统
- 自动从外部网页抓取并入库
- 复杂权限系统

## 3. 当时的关键设计判断

### 3.1 不把资料继续塞进 `topic`

当时的判断是：

- `topic` 继续表达“要辩什么”
- 参考资料应作为与辩题并行的独立输入通道存在

### 3.2 不把大块文档直接塞进会话主快照

规划阶段就明确担心：

- 会话读取响应会快速膨胀
- 前端打开历史会话时 hydrate 负担过大
- 每次保存状态都会重复写入大 JSON
- 文档重处理难以局部更新

### 3.3 先做结构化压缩，不做复杂检索

当时选择的是：

- 一份文档保留有限摘要和高价值条目
- 先控制 token 消耗
- 不在 MVP 阶段引入向量库和复杂索引
